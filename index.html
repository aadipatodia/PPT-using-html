Three-agent design

Intent → action → conversation guard: understand the ask, execute safely, keep the session coherent.

Outcome: higher accuracy, fewer wrong actions, better user experience.

Agent 1

Intent brain

  • Uses Gemini to classify messages into supported intents (assignment, update, performance, users, and related scopes).
  • Handles ambiguity with targeted clarification—for example, “pending tasks” for self vs team.
  • Stays within task-management scope before deeper processing.
  • Emits confidence and reasoning for downstream flow control.
Agent 2

Action brain

  • Uses policy-driven prompting to extract and normalise required fields per intent.
  • Keeps multi-turn state until all required values are captured; minimal clarification when something is missing.
  • Calls backend tools and Appsavy APIs only when data is complete.
  • Returns structured, professional responses to the user.
Agent 3

Guard brain

  • Detects topic shifts and new requests mid-conversation.
  • Prevents mixing old and new requests or unrelated actions in one turn.
  • Handles confirmation-stage interruptions without losing context; on shift, asks to continue prior context or start fresh.
  • Reduces execution errors in real-world chat behaviour.

Appsavy connection

  • Appsavy is the business backend: task, user, and report APIs.
  • The AI layer stays thin—validated payloads only—and sends them to Appsavy endpoints.
  • Operations include create/update tasks, status changes, counts and reports, and user operations.
  • Enterprise workflow stays centralized in Appsavy; WhatsApp is the UI, Appsavy is the system of action.

Gemini connection

Language understanding and reasoning across the stack—intent, parameters, clarification, and intent-shift (Agent 3)—plus routing hints and context boundaries.

  • Wrapped with timeout and fallback for reliability.
  • Turns natural language into machine-actionable data for downstream agents and APIs.

Outputs feed agents; storage stays in Appsavy / DBs

Redis + MongoDB connection

Redis for fast conversation state; MongoDB for persistent user and org truth.

Redis — real-time memory / state

  • Session history for multi-turn conversations
  • Deduplication of repeated webhook events
  • Temporary pending states, locks, and conversation continuity

MongoDB — identity + org layer

  • Registered users and phone / user mapping
  • Role resolution (manager / employee)
  • Team hierarchy lookup for authorization and routing

Identity checks from MongoDB run throughout the flow alongside Redis-backed session state.

End-to-end system flow

From inbound message to WhatsApp reply, with Redis, Gemini agents, Appsavy, and MongoDB in the loop.

  1. WhatsApp message → webhook → dedup (Redis)
  2. Agent 1 classifies intent (Gemini)
  3. Agent 3 checks context shift (Gemini + Redis history)
  4. Agent 2 extracts fields and validates readiness (Gemini + policy)
  5. Appsavy API executes the business action
  6. Response is sent back to WhatsApp
  7. User / team identity checks from MongoDB throughout
